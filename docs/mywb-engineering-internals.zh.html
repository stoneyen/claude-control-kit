CLAUDE.md · 解剖

操作手冊的結構

它是 agent 每次 session 最先載入、且被要求遵守的檔。段落不是章節,是行為規約:

段落作用(對 agent)
How ADRs & specs are recorded ⚠️四個 .dev/ 源頭是真相;改前先查。含 ADR 索引表。
Avoiding test drift ⚠️每個行為變更帶測試;哪種改動要哪種測試。
Architecture: CA + DDD分層非妥協項、Money-as-Decimal、/api/ router 規則。
Session / shell habits「每個 until 迴圈要 timeout」「同 aggregate 不存兩次」。
Commands怎麼跑測試/lint/部署 —— agent 直接照抄。

設計要點

① 可 grep:ADR 索引一行一決策,agent 秒查「這改動牴觸哪條決策」。

② 祈使句:每行回答「該做/不該做什麼」。純描述結構的行是噪音。

③ ⚠️ 標記硬性規則,和一般指引區分。

④ 自我維護:CLAUDE.md 是 strict-review 觸發項,漂移了就 docs-sync。
ADR · 檔案設計

一決策一檔,用號碼串起推理鏈

- Status: Accepted | Proposed | Superseded

## Context      # 當時的取捨、為什麼要決定

並行工作的 ⚠️ 陷阱:撞號

多個 session 各自取「下一個號」→ 兩份 ADR-064。這是真實踩過的。
解法慣例:先提交的保留原號;耦合較深(被 code 引用多)的保留原號,另一份改到空號並更新全部引用。CLAUDE.md 索引裡留一則「已解決」註記。
更好的預防:就是 worktree 隔離(後面 §)—— 減少並行同 tree 取號。

script · session-registry.sh

co-tenant 偵測 = 檔案式心跳

# 每 session 一個檔,內容 = 它所在的 worktree 頂層
WT_REGISTRY="${WT_REGISTRY:-${TMPDIR:-/tmp}/claude-session-registry}"
WT_TTL="${WT_TTL:-600}"   # 秒;安靜這麼久 = 死掉

sr_register() {   # 刷新本 session 心跳
  local sid="$1" top="$2"; [ -n "$sid" ] || return 0
  mkdir -p "$WT_REGISTRY"
  printf '%s' "$top" >"$WT_REGISTRY/$sid"
}
sr_cotenants() {  # 數同一棵樹的其他 live session
  local sid="$1" top="$2" now n=0; now="$(date +%s)"
  for f in "$WT_REGISTRY"/*; do
    [ "$(basename "$f")" = "$sid" ] && continue
    age=$(( now - $(mtime "$f") ))
    [ "$age" -gt "$WT_TTL" ] && { rm -f "$f"; continue; } # 剪掉死的
    [ "$(cat "$f")" = "$top" ] && n=$(( n + 1 ))
  done
  echo "$n"
}

為什麼這樣設計

無 daemon、無鎖伺服器 —— 只是 $TMPDIR 下的檔,mtime 當心跳。
自我修剪:超過 TTL 的檔在數的時候順手刪。
心跳在最忙的 hook 刷新(每次 Bash 呼叫)→ 免定時器就有好頻率。
key 是 session_id(hook JSON 帶),value 是 worktree 頂層 → 「co-tenant」= 同頂層的其他 session。

只擋會互踩的操作,且只在有 co-tenant 時

# 讀 hook JSON
sid=$(jq -r '.session_id')  cmd=$(jq -r '.tool_input.command')

[ "${WT_ALLOW_SHARED_TREE:-0}" = 1 ] && exit 0  # 逃生門

is_dangerous() {   # 一個 segment 內比對,忽略 `git log`
  # checkout/switch 但排除 `-- <file>` 還原
  grep -qE 'git[^;&|]*(checkout|switch)' && \
    ! grep -qE 'git[^;&|]* -- ' && return 0
  grep -qE 'git[^;&|]*stash' && return 0
  grep -qE 'git[^;&|]*reset[^;&|]*--hard' && return 0
  return 1
}
is_dangerous || exit 0
[ "$(sr_cotenants "$sid" "$top")" -ge 1 ] || exit 0  # 單人 → 放行
echo "🚫 BLOCKED: 共用樹切分支/stash" >&2; exit 2

設計取捨(誠實)

只擋破壞性操作:branch switch / stash / reset --hard;git checkout -- file(還原單檔)放行。
單人零成本:sr_cotenants ≥ 1 才擋。
已知誤報:regex 會誤中「字串裡提到 git checkout」(如 echo "git checkout")。保守 = 安全;WT_ALLOW_SHARED_TREE=1 一次性繞過。
script · gate-runner.sh

可插拔的 commit gate — runner 與規則分離

# 由 .githooks/pre-commit 呼叫(git hook 沒接好時由 Claude fallback)
STAGED_FILES="$(git diff --cached --name-only \
                 --diff-filter=ACMR)"
export STAGED_FILES
[ -n "$STAGED_FILES" ] || exit 0          # 空 stage → 過
[ -d "$GATES_DIR" ] || exit 0

rc=0
for gate in "$GATES_DIR"/*; do            # .dev/gates/* 依序
  [ -x "$gate" ] || continue
  case "$gate" in *.sample|*.md) continue;; esac
  # staged 檔清單餵 stdin;gate 非 0 = 擋
  printf '%s\n' "$STAGED_FILES" | "$gate" || {
    echo "🚫 gate failed: $(basename "$gate")" >&2; rc=1; }
done
exit "$rc"

合約(你的 gate 要遵守)

① staged 檔清單從 stdin 進(也在 $STAGED_FILES)。
② exit 0 過、非 0 擋,stderr 顯示。
③ 數字前綴排序:10-lint.sh → 40-arch.sh。
④ .sample/.md 忽略(複製 sample → 改名 → chmod +x)。
runner 語言無關;規則是專案的。 harness 只負責接線與排序。
script · pretooluse-bash.sh + pre-commit

commit gate 的雙重保險

# pretooluse-bash.sh — Claude PreToolUse(Bash) 派發器
is_git_commit && {
  hp="$(git config --get core.hooksPath)"
  # git 的 pre-commit 已生效 → 交給它,避免雙跑
  [ "$hp" = ".githooks" ] && exit 0
  # 否則 fallback:現在就跑 gate,擋住
  gate-runner.sh || exit 2
}
is_git_push && grep -q 'main' && \
  echo "⚠️ 直推 main — 建議走 PR" >&2   # 只警告

兩條路,同一個 gate:

  • 主要:git pre-commit 對所有 commit 生效(人 + agent),前提是設了 core.hooksPath=.githooks。
  • fallback:若沒設,Claude 的 Bash hook 偵測到 git commit 就代跑 gate 並 exit 2 擋下。
commit-scoped(參考專案版):只在 commit stage 了 backend/src 時跑重的靜態 gate(ruff/arch/spec-sync/mypy);純 docs commit 秒過,不被無關的分支漂移拖累。pytest/tsc 留給 CI(每 commit 跑太慢)。
script · validate_arch.py

架構 lint = AST 走訪 + 22 條規則

被 _WATCHED 命中的變更 source,同 diff 裡必須有對應 test 變更 —— 除非它也命中 _EXEMPT。三種「有對應」的判定:

scoping(細節但重要)

--base main 時只看 committed range + staged —— 未 stage 的編輯、未追蹤檔(常是別的並行 session 的 WIP)排除掉,才不會被別人的漂移擋住。

.claude/settings.json · 接線

hook 怎麼綁到工具事件

{
  "hooks": {
    "PreToolUse": [
      { "matcher": "Bash", "hooks": [
        { "type":"command",
          "command":"bash \"${CLAUDE_PROJECT_DIR}/.claude/hooks/pretooluse-bash.sh\"",
          "timeout":200 },

          "command":"bash \"…/worktree-guard.sh\"",

${CLAUDE_PROJECT_DIR} 讓路徑與 clone 位置無關。

script · claude-worktree.sh

per-session worktree launcher

# 用 `cw [slug]` 取代 `claude`
REPO="${WT_REPO:-$(git -C "$_here" rev-parse --show-toplevel)}"
WT="${WT_ROOT}/${SLUG}-$(date +%Y%m%d-%H%M%S)-$$"
BRANCH="wt/${SLUG}-…-$$"

git -C "$REPO" worktree add "$WT" -b "$BRANCH"  # 全新樹
( cd "$WT" && claude "${@:2}" ); rc=$?

# 退出時:乾淨(無未提交、無未推)才自動移除
dirty="$(git -C "$WT" status --porcelain)"
unpushed="$(git -C "$WT" log --branches --not --remotes)"
if [ -z "$dirty" ] && [ -z "$unpushed" ]; then
  git -C "$REPO" worktree remove "$WT" --force
  git -C "$REPO" branch -D "$BRANCH"
else
  echo "▸ KEPT(有未提交/未推): $WT" >&2
fi

安全設計

隔離 by construction:每 session 一棵樹,天生不互踩 —— guard hook 只是安全網。
絕不默默丟工作:退出時只有乾淨(無未提交 + 無未推)才移除;否則保留並印路徑。
從主樹當下 HEAD 分岔 → 主樹保持在最新 main 最理想。

CI · 部署 gate 的 YAML

image build 接在 checks 之後的實作

# backend-image.yml — 不再 on: push
on:
  workflow_run:
    workflows: ["checks"]
    types: [completed]
    branches: [main]    # PR 的 checks(非 main head)排除
  workflow_dispatch:    # 保留手動逃生門
jobs:
  build-push:
    if: >
      github.event_name == 'workflow_dispatch' ||
      github.event.workflow_run.conclusion == 'success'
    env:
      TRIGGER_SHA: ${{ github.event.workflow_run.head_sha }}
    steps:
      - run: |
          SHA="${TRIGGER_SHA:-$GITHUB_SHA}"
          git checkout "$SHA"       # checks 通過的那個 commit
          # workflow_run 無 paths: → job 內自己過濾
          git diff --name-only "${SHA}^" "$SHA" \
            | grep -qE '^backend/' || { echo skip; exit 0; }

三個非顯而易見的細節

① 用 workflow_run.head_sha checkout —— 否則會拿到 default branch head,不是 gate 通過的 commit。
② workflow_run 沒有 paths: → 在 job 內用 git diff SHA^ SHA 重建 path 過濾。
③ 首次生效在 merge 後:workflow_run 永遠用 default branch 的 workflow 檔。

#!/usr/bin/env bash
set -uo pipefail
staged="$(cat)"          # stdin = staged 檔清單
js="$(printf '%s\n' "$staged" | grep -E '\.(ts|tsx)$')"
[ -n "$js" ] || exit 0   # 沒動到 → 過
node_modules/.bin/eslint $js   # 非 0 = 擋 commit

gate 就是一支讀 stdin、用 exit code 表態的可執行檔。要加什麼檢查,就寫一支丟進 .dev/gates/。

  • 只查 staged 的相關檔 → 快;不相干的 commit 秒過。
  • 命名排序 決定執行順序。
  • 重的檢查放 .dev/gates/push/(pre-push 跑)或 CI。
  • 驗證方式:故意違規 → 應擋;修好 → 應過。

Python/DDD 專案可直接 --profile python-ddd 拿到 ruff/mypy/arch/spec-sync/test-drift 六支現成 gate。

貫穿所有 script 的設計原則

為什麼每支都這樣寫

  • idempotent —— installer 跳過既有檔;gate 可重跑;心跳可覆寫。
  • fail-loud —— adapter 缺依賴就在啟動時炸;不默默 fallback 到 stub。
  • 非破壞 —— 不覆蓋既有 settings.json/CLAUDE.md;guard 不擋單人。
  • staged-scoped —— gate 只看這次 commit 的檔,不被無關漂移拖累。
  • exit code 是介面 —— 0 過 / 2 擋 / stderr 回饋;語言無關。
  • env 參數化,不改邏輯 —— CCK_PKG / WT_* 讓通用腳本適配專案。
  • 逃生門明示 —— WT_ALLOW_SHARED_TREE=1 / --no-verify / workflow_dispatch。
  • 可攜 vs 專屬分離 —— runner/wiring 通用,規則放 .dev/gates/ 由專案填。
