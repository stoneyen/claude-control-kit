問題

AI 寫程式很快。難的是讓系統長期一致。

生成速度暴露了真正的瓶頸:如何在時間、多人、如今還有多個 agent 之下維持一致。這些是我們一再踩到的失效模式:

Spec 漂移

程式改了,設計文件默默落後。下一個讀的人相信了一個謊。

Test 漂移

行為變了,測試留在舊版。套件仍全綠,卻不再描述程式碼。

CI 綠 ≠ 可部署

image build 與測試 gate 並行 —— 紅燈照樣把壞掉的 code 推上 prod。

並行互踩

兩個 agent session 共用一棵 working tree;一個 git checkout 把另一個未提交的工作連根拔起。

決策無紀錄

「為什麼這樣做?」每隔幾週就被重新爭論、或不小心被破壞一次。

教訓流失

同一個非顯而易見的 bug 從頭再查一次 —— 本來 10 分鐘的修法又花掉一天。

這份手冊裡的每一條規則,都是因為上面某一項真的咬過我們一次。沒有一條是空談。

核心主張

讓 agent 協作開發耐久的四個動作

1

源頭放進 repo

決策、行為、血淚 ops 知識、工作狀態 —— 版本控制在程式碼旁,而非會腐化的 wiki。(.dev/)
2

給 agent 一本操作手冊

一份 CLAUDE.md,是 agent 必須遵守的指令 —— 慣例、決策索引、spec-first 規則。不是文件,是指令。
3

用 harness 在 CI 之前強制

hooks + git gate 在最早的邊界就阻擋(edit → commit → push),CI 才查的紀律在你推之前就先擋下。
4

隔離並行

每個 session 一棵 git worktree,並行的人/agent 永遠無法破壞彼此的分支或未提交狀態。

接下來就是把這四點講具體 —— 最後是怎麼裝進你的專案。

動作 1 · 源頭

.dev/ 四個目錄

四種耐久知識,各一個目錄。「做任何變更或回答設計問題前,先查這些。」

目錄放什麼紀律
.dev/adr/架構決策紀錄,一決策一檔(Context · Decision · Consequences · Alternatives)。改架構前先 grep。牴觸某條?對齊它,或寫新 ADR supersede 舊的。絕不默默違反。
.dev/specs/每個 aggregate 的行為 spec:entity spec + 每個 use case 一份 YAML(invariants、pre/postconditions、events、tests)。寫程式前先與人 review。spec-first。
.dev/lessons/血淚修法,以症狀命名。查非顯而易見的 bug 前先 grep;修好後寫一則。
.dev/tasks/檔案式 task ledger —— 跨 session、跨當機都存活。跨 session 回答「上次做到哪?」。

wiki 在旁邊、沒版本控制。這些就在程式碼旁,隨每次 clone 與 PR 一起走。

源頭 · 決策

ADR —— 撐過人員流動的決策

  • 一決策一檔,編號:ADR-065-checks-gate-blocks-deploy.md。
  • 固定結構:Context(當時的取捨)· Decision · Consequences · Alternatives Considered · Related。
  • 用 supersede,不刪除。新決策寫新 ADR,並指名它取代的那條 —— 推理鏈完整保留。
  • CLAUDE.md 裡有一張索引表,每條決策一行摘要供 agent grep。(目前 75 條、還在增加。)

回報:「為什麼長這樣?」幾秒內有答案;會破壞某決策的改動,在出貨前就被擋下。

# ADR-065 — The checks gate blocks deployment
- Date: 2026-07-05
- Status: Accepted

## Context
image build 與 checks 在 push 時並行,
紅 gate 照樣出貨。此 repo 方案的
branch protection 回 403。

## Decision
產出部署的 workflow 改由
workflow_run(checks 成功)觸發,而非 push。

## Consequences
紅 checks → 無 image → 無部署。
部署延遲 +一次 checks。

## Alternatives Considered
- Branch protection(方案不支援,403)— 否決。
- 並行 + 事後 rollback — 被動,已出貨。
源頭 · 行為

Spec-driven development(規格先行)

變更的設計在寫程式之前先寫、先 review。再用機器讓 spec 與 code 不漂移。

  • spec-first 規則:要編輯 domain aggregate / use case / HTTP route 前,先更新它的 spec —— 而且是硬 gate:新增的 aggregate/UC 若沒有對應 spec 目錄,直接擋 commit。
  • 兩份產物:entity spec(結構、invariants、狀態機)+ 每個 use case 一份 YAML(input、pre/postconditions、events、test scenarios)。
  • check_spec_sync 斷言 entity spec 與 use-case YAML 保持一致 —— 程式改了欄位名但 spec 沒跟,gate 就紅。
  • 有一條引導流程(/execute-uc)從 review 過的 spec 產出骨架,再跑四道 gate。

為什麼贏過「先寫 code、之後補文件」

spec 是人核准的契約。agent 是照 spec 實作,不是照感覺。而因為有 checker 綁 spec↔code,文件無法悄悄變成虛構 —— 那正是多數文件不可信的原因。
CLAUDE.md 裡的觸發語(原文): 「若你即將編輯 domain/、application/、routers/ 底下任何檔案 —— 停。先確認對應 spec 是否存在。沒有,就先寫 spec。」
源頭 · ops 知識

Lessons —— 別再重複踩同一支耙子

每個非顯而易見的 bug 一份 markdown,以症狀命名讓下一個人 grep 得到。成本約 10 分鐘;省下的是下一個人不再損失一天。

tilt-live-update-silently-fails

tilt trigger 回報成功、pod Running,但 pod 裡的 code 是舊的。

ci-green-but-deploy-ships-broken

紅 gate 下 image 照建照部署 —— workflow 並行。

runner-drops-builder-label-on-restart

docker 重啟讓 runner 用舊 env 重新註冊 → image job 永久排隊。

結構:何時會咬 · 症狀 · 根因 · 修法 · 如何提早偵測 · 相關。 規則落成常規:修好一個非顯而易見的 bug 後,寫下 lesson 並從 commit 連結它。

動作 2 · 操作手冊

CLAUDE.md 是指令,不是文件

這是 agent 最先讀、且被要求遵守的檔。它不描述系統 —— 它告訴 agent 在系統裡該怎麼行動。

  • ADR 索引 —— 每條決策一行、可 grep。
  • 不可妥協事項 —— 分層規則、「錢一律用 Decimal,絕不 float」、「router 註冊在 /api/<resource>」。
  • spec-first 觸發 —— 一個明確的硬性停止點。
  • enforcement hook 表 —— 什麼在哪裡擋。
  • session 習慣 —— 「每個輪詢迴圈都要有 timeout」、「同一個 aggregate 不要存兩次」。

它要通過的檢驗

每一行都要回答:「agent 在這裡該做/不該做什麼?」如果只是在描述程式碼本身就看得到的結構,那是噪音 —— 刪掉。

被同一套 harness 盯著

CLAUDE.md 本身就是 strict-review 觸發項。當它與現況漂移(像並行工作 5 天後那樣),docs-sync 就是一等任務 —— 這個 session 的歷史裡就正在做這件事。
實例 · 把架構變成 gate

Clean Architecture + DDD,機器強制

# src/<pkg>/  — 依賴只能向內
domain/          純 Python:entities、value
                objects、repo ports、events
                ✗ 不可 import sqlalchemy/fastapi

「domain 不 import 框架」、「錢一律 Decimal」、「DTO 在邊界」這些不只是 review 口耳相傳 —— 一個 linter 在每次動到 source 的 commit 上強制其中 22 條。

  (此層禁止)

可攜性:規則本身就是架構 —— linter 只參數化 package 名。一個「同型」專案把 CCK_PKG 指向自己的 package,就套用同樣 22 條規則。
動作 3 · enforcement harness

把 gate 左移 —— 在最早的邊界就擋

CI 在 PR 邊界擋。這些 hooks 把同樣的問題在你還沒推之前就抓到 —— 便宜到昂貴,各自在自己的邊界阻擋。

位置跑什麼會擋?
每次編輯 · PostToolUselint 剛編輯的檔回饋 當下修
編輯 domain/UC/router 檔 · PreToolUsespec-first 提醒警告
git commit · .githooks/pre-committest-drift · spec 存在性 ·(動 source 時)lint · arch · spec-sync · types是 — 阻擋
git push · .githooks/pre-pushtest-drift vs base 分支是 (--no-verify 可繞)
共用樹互踩 · PreToolUse有 co-tenant 時的 git checkout/switch/stash/reset --hard是 (單人不受影響)
PR · CI checks workflow完整 pytest · mypy · tsc -b · vitest是(server 端)

同一套紀律兩種實作:git hooks 擋人也擋 agent;Claude Code hooks 加上 agent 專屬守衛,以及 git hooks 沒接好時的 fallback。

值得單獨點名的 gate

test-drift:每個行為變更都帶一個測試

被監看的 source 檔改了、同一個 diff 裡沒有對應的測試變更 → 擋 commit。它把每個 source 檔對應到它的測試 —— 用 stem、用 router 的 fuzz 覆蓋、或用 bounded-context 群組。

  • 刻意可豁免:純重構、plumbing、migration、常數 —— 但你要在 commit 說明。重點是每次變更都是一個有意識的決定,而非默默漂移。
  • 在 commit(staged)、push(vs base 分支)、CI(commit-range)各跑一次。
# pre-commit
== test-drift guard ==  base=main
  ✅ stem  application/place_order.py
  ❌ none  domain/order.py

⚠️  1 個 source 變更沒有測試變更:
    - domain/order.py

🚫 commit 被擋
加個測試,或在 commit 訊息宣告
這是 test-neutral。
動作 4 · 隔離並行

每個 session 一棵 git worktree

血淚故事

兩個 agent session 共用一棵 working tree。一個 git checkout 切到別的分支 —— 另一個 session 未提交的工作就散落到錯的分支、被吞進 stash。復原花了整整一次 worktree 重建。

分層修法:一個 launcher 讓每個 session 都在全新 git worktree 啟動(乾淨就自動移除)—— 預設隔離。再加一個 guard hook 當安全網。

# guard 只在有 live co-tenant 時觸發
$ git checkout main
🚫 BLOCKED:共用樹裡切分支。
此樹另有 1 個 live session。
  切分支會把他們未提交的工作拔掉。
改用獨立 worktree:
  git worktree add ../repo-x -b feature-x
# 單人?完全不受影響。一次性繞過:
#   WT_ALLOW_SHARED_TREE=1
心跳登記:每個 session 觸碰一個檔、記錄它在哪棵樹。「co-tenant」= 同一棵樹裡另有 live session。單人流程零成本。
CI/CD · gate

checks workflow

GitHub Actions,跑在 self-hosted runner。一個叫 checks 的 workflow 就是整個品質 gate —— 本地 hooks 強制的同一套紀律,在 server 端權威地再跑一次。

backend

ruff · mypy · validate_arch · check_spec_sync · test-drift · pytest(記憶體 sqlite,無 sidecar)

web

tsc -b(project refs 含測試 —— 不是 --noEmit)· vitest

web-perf

對 production build 跑 Lighthouse budget —— 大 dep 進入口 chunk 就變紅

本地 gate 做的每件事,CI 都以真相來源再做一次 —— 就算本地 hook 被繞過,紅 gate 仍擋得住 merge。

CI/CD · 部署閘門(ADR-065)

「CI 綠」必須真的擋住部署

微妙的失效:image build 與測試 gate 在 push 時並行。紅 gate 照樣建 image、照樣出貨。CI 看起來綠、prod 卻壞了。

之前 —— 兩者競速
push mainchecks  +  image-build 並行
↓
flux部署 image —— 即使 checks 紅了
之後 —— image 接在 checks 之後
push mainchecks 執行
↓ workflow_run · conclusion==success
image-build此時才建 — flux 才部署

紅 checks → 無 workflow_run success → 無 image → 無部署。 build 會 checkout gate 通過的那個 commit。不需要 branch-protection 方案。

CI/CD · 交付

GitOps + immutable image

  • Flux 把 git → cluster 收斂。部署狀態就是一個 commit;一次 bump 是 chore(gitops): bump images commit,可 review、可 revert。
  • 每個部署的 pod 都 pin 不可變 ref(digest / git-sha)。沒有可變的 :dev/:latest —— 「rollout restart」無法默默服務舊 code。
  • 一個 runtime guard(verify-deployed-code.sh)斷言 running == intended:pod 的 source hash 對得上工作樹。綠燈代表 cluster 真的跑你最新的 code。

為什麼「immutable + verify」

當 tag 不會在你腳下變、又有腳本證明實際跑的是什麼,一整類「我這邊沒問題 / pod 是舊 code」的 bug 就消失了。這個 repo 有兩則 lesson 正是因為可變 tag 說了謊。
部署後 smoke 對 live pod 再驗一次(真的 cookie 認證、schema-fuzz 抓 5xx)—— 「已上線且驗證」之前的最後一道防線,誠實地說出來。
CI/CD · 驗證的層次

六種測試,各抓不同的失效

不是「有測試就好」,而是每一層攔一種不同的問題 —— 由快而慢、由窄而寬。

層抓什麼何時跑
unit領域邏輯 / value object / invariants(含 Hypothesis property)commit · CI
integrationUC + repo + adapter 串起來 —— 用記憶體 sqlite,免 sidecarCI
fuzz(schemathesis)每個 GET endpoint 對亂數輸入不 5xxCI(in-process)· 部署後(對真 stack)
e2e(Playwright · 6 specs)SPA 端到端流程(testid 斷言,不綁死值)CI · 部署後 smoke
web-perf(Lighthouse)效能預算 —— 大 dep 進入口 chunk 就紅CI(advisory)
smoke(post-deploy)對線上 pod 真 cookie 認證 + schema-fuzz 抓 5xx部署後,自動

關鍵:測試都跑無外部服務的記憶體 sqlite → 快、穩、可平行;真 stack 的驗證留給部署後 smoke。

CI/CD · code review + code scan

人看不到的,交給機器與多模型

異質 LLM code review

  • /multi-review —— 把 git diff main...HEAD 同時丟給 三家(Claude · Gemini CLI · Codex CLI)並行 review,再綜合成跨模型一致度報告。單一模型的盲點,另兩家常補到。
  • strict-review 觸發:diff 動到 aggregate/UC/router/migration/ADR 時,自動建議先跑 review 才開 PR。
  • /security-review —— 對變更做專門的安全審查。

5 道 code scan(security.yml)

掃描gate
gitleaks(祕密外洩)BLOCK 任何祕密
pip-audit / npm audit(依賴 CVE)只擋 CISA KEV 其餘 warn
bandit(Python SAST)WARN

KEV-gating 的巧思:只在「實際被利用中」的 CVE 擋,其餘警告 —— 不讓弱點 noise 卡死每次 merge。

DevOps · 機密設計(Vault)

祕密:一把 KEK、不進該進的 pod、進 git 也加密

① 一把 KEK 加密資料

Fernet 對稱加密,單一 KEK(base64 32-byte)。KEK 從 k8s Secret 經 envFrom 注入成 MYWB_PASSWORD_VAULT_KEK —— 禁止從 .env / cmdline 讀。明文一律包 SecretStr 防誤 log。

② 祕密不進 agent pod

Warren runtime pod 看不到 KEK/OAuth/broker 憑證。資料操作走 per-session 短命 token callback 回 api(api 才持 KEK + DB)。einvoice-scraper 同理:Google-blind,憑證每次請求傳。

③ GitOps 祕密加密在 git

SOPS + age 把 secret manifest 加密後放進 git(可版本控制、跨機同一把 KEK);Flux 用 age key 於叢內解密。祕密不再散落 out-of-band。

身分:Google OAuth 登入 + server 發的 session cookie(DB 只存 token 的 SHA-256);X-User-Id header 僅 dev fallback,prod 關閉(否則公開 funnel 上任何人可冒充)。

整體如何串起來

從想法到運行的 code,每一步都有 gate

1 · spec先寫 + review spec
→
2 · code在獨立 worktree 編輯
→
3 · committest-drift · arch · types · spec-sync
→
4 · push / PR完整 CI checks
5 · mergemain 上的 checks
→
6 · buildchecks 綠才建 image
→
7 · deployflux · 不可變 ref
→
8 · verifyrunning==intended + smoke

注意這個形狀:同一批規則(spec-first、test-drift、arch)在編輯時、commit 時、CI 裡各查一次。刻意重複 —— 每一層都是上一層的安全網。

一個具體例子 · 跟著一個變更走一遍

「幫 CardAccount 加一個 credit_limit 並在 API 顯示」

同樣一件小事,在這套 harness 下開發者實際會遇到什麼 —— 每一處「阻力」其實都是護欄。

  1. 開 session → cw add-credit-limit → 自動落在獨立 worktree,不碰別人的樹。
  2. 想直接改 domain/card_account/entity.py → spec-first 提醒跳出 先寫 spec
  3. 寫 spec + 與 owner review:entity spec 的 Attributes 表加一列、update-card.yaml 的 inline schema 跟上。
  4. 實作:domain 用 Money(不是 float)、UC、router 的 CardResponse 加欄位、補一個測試。
  5. git commit → commit gate:忘了測試 → test-drift 擋 · spec 表與 yaml 對不上 → spec-sync 擋 · 用了 float → arch 擋。修好才過。
  6. (若沒開 worktree、又有並行 session)git checkout 切分支比對 → worktree guard 擋,叫你開 worktree。
  7. push → PR → CI 完整跑一次(pytest · tsc -b · vitest)。
  8. merge → main 的 checks 綠 → image build → Flux 部署 → verify-deployed-code.sh + smoke 確認線上真的是這版。

重點體驗:每個 gate 都在你還沒犯錯太遠時就攔下,而且訊息直接告訴你怎麼修 —— 阻力即護欄。

每條規則都有來由

規則 ← 真實事故

發生了什麼它催生的規則
兩個 session 共用一棵樹;一個的 git checkout 把另一個的工作打散。每 session 一 worktree guard + launcher。
紅測試 gate 下 image 照建照出貨;prod 跑了 CI 沒過的 code。image build 接在 workflow_run(checks 成功)之後。
一個 endpoint 在 prod 500,因為觀測 wrapper 在某路由崩潰 —— 單元測試因 wrapper 關閉而通過。用與 prod 完全相同的 middleware 堆疊;測真實的 ASGI 介面。
行為變更帶著舊測試出貨;套件仍全綠。test-drift gate。
一份密集帳單重新解析,逐一觸發所有 pipeline 大小上限 —— 一個檔 8 個 commit。每個 stage 都以最壞情況設容量;寫一則 lesson 讓下一次只要 1 個 commit。

這才是重點:別在前期把治理設計完 —— 讓每次事故硬化成一個 gate 或一則 lesson。 harness 就是它們長出牙齒的地方。

開始上手

採用檢查清單

  1. 在你的 repo 跑 install.sh(合適的話加 --profile python-ddd)。冪等、非破壞。
  2. 啟用 git hooks: git config core.hooksPath .githooks。把 hooks 區塊 merge 進 .claude/settings.json。
  3. 填 .dev/gates/ 你真正的 lint / typecheck / test-drift / arch gate。確認每個在故意違規時擋、修好後放行。
  4. 從模板寫你的 CLAUDE.md —— stack、慣例,並開一個 ADR 索引(ADR-000 是模板)。
  5. 接 CI:一個 checks workflow;若你 push 就部署,採用 workflow_run 部署 gate 讓紅 gate 擋住交付。
  6. 加 cw launcher 到你的 shell rc;用它開 session,每個都拿到獨立 worktree。
  7. 然後讓它長:每次事故 → 一個新 gate 或一則 lesson。系統會自我硬化。
重點帶走

七條可移植原則

  • 源頭放進 repo —— 決策、行為、lessons、工作狀態。版本控制、可 grep、在程式碼旁。
  • 給 agent 一本操作手冊 —— 它必須遵守的指令,而非它可能會讀的散文。
  • spec-first —— 設計在寫程式前先 review,再用 checker 讓兩者不漂移。
  • enforcement 左移 —— 在 edit/commit/push 就擋,不只在 PR。刻意多層冗餘。
  • 每個行為變更都帶測試 —— 沒有默默的 test drift。
  • CI 綠必須擋住部署 —— build 接在 gate 之後,而非 push 之後。
  • 隔離並行 —— 每 session 一 worktree;並讓每次事故硬化成 gate 或 lesson。
拿去用

目標不是規則。是讓系統 —— 以及觸碰它的 agent 數量 —— 成長時仍保持一致。
