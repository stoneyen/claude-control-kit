The problem

AI writes code fast. Keeping a system coherent is the hard part.

Speed of generation exposes the real bottleneck: staying consistent across time, many hands, and now many agents. These are the failure modes we kept hitting:

Spec drift

Code changes; the design doc silently falls behind. The next reader trusts a lie.

Test drift

Behaviour changes; its test is left stale. The suite stays green but no longer describes the code.

Green CI ≠ safe deploy

Image builds raced the test gate — a red gate still shipped broken code to prod.

Concurrent stomp

Two agent sessions share one working tree; one git checkout rips the other's uncommitted work out.

Undocumented decisions

"Why is it built this way?" gets re-litigated or accidentally broken every few weeks.

Lost lessons

The same non-obvious bug is rediscovered from scratch — the 10-minute fix costs a day again.

Every rule in this playbook exists because one of these actually bit us. None are aspirational.

The thesis

Four moves that make agent-driven dev durable

1

Sources of truth live in the repo

Decisions, behaviour, hard-won ops knowledge, and work state — versioned next to the code, not in a wiki that rots. (.dev/)
2

An operating manual for the agent

A single CLAUDE.md of instructions the agent must follow — conventions, the decision index, the spec-first rule. Not docs; instructions.
3

A harness that enforces it — left of CI

Hooks + git gates that block at the earliest boundary (edit → commit → push), so the same discipline CI checks is caught before you even push.
4

Isolate concurrency

One git worktree per session so parallel humans/agents can never corrupt each other's branch or uncommitted state.

The rest of this deck is these four, concretely — then how to install them in your project.

Move 1 · Sources of truth

The .dev/ directories

Four kinds of durable knowledge, each a directory. "Always check these before making changes or answering design questions."

DirHoldsThe discipline
.dev/adr/Architecture Decision Records — one file per decision (Context · Decision · Consequences · Alternatives).Before an architectural change, grep here. Contradict one? Align, or write a new ADR that supersedes it. Never silently violate.

A wiki is off to the side and unversioned. These sit next to the code and travel with every clone and PR.

Sources of truth · decisions

ADRs — decisions that survive turnover

  • One file per decision, numbered: ADR-065-checks-gate-blocks-deploy.md.
  • Fixed shape: Context (the forces) · Decision · Consequences · Alternatives Considered · Related.
  • Supersede, don't delete. A new decision writes a new ADR that names the one it replaces — the reasoning chain stays intact.
  • An index table in CLAUDE.md gives the agent a one-line summary of every decision to grep. (75 and counting.)

The payoff: "why is it like this?" is answerable in seconds, and a change that would break a decision is caught before it ships.

# ADR-065 — The checks gate blocks deployment
- Date: 2026-07-05
- Status: Accepted

## Context
Image builds raced `checks` on push to main,
so a red gate still shipped. Branch protection
returns 403 on this repo plan.

## Decision
Deploy-producing workflows trigger on
workflow_run of `checks` succeeding — not push.

## Consequences
Red checks → no image → no deploy.
Deploy latency +one checks run.

## Alternatives Considered
- Branch protection (403 on plan) — rejected.
- Parallel + post-hoc rollback — reactive.
Sources of truth · behaviour

Spec-driven development

The design of a change is written and reviewed before the code. A machine keeps the spec and the code from drifting apart.

  • Spec-first rule: before editing a domain aggregate, use case, or HTTP route, update its spec first — and it's a hard gate: a new aggregate/UC with no spec directory blocks the commit.
  • Two artefacts: an entity spec (structure, invariants, state machine) + one YAML per use case (input, pre/postconditions, events, test scenarios).
  • check_spec_sync asserts the entity spec and the use-case YAML stay in agreement — a renamed field in code with a stale spec fails the gate.
  • A guided flow (/execute-uc) generates the skeleton from the reviewed spec, then runs the four gates.

Why this beats "write code, doc later"

The spec is the contract the human approves. The agent implements to the spec, not to a vibe. And because a checker binds spec↔code, the doc can't quietly become fiction — the failure mode that makes most docs untrustworthy.
Trigger, verbatim from CLAUDE.md: "If you are about to edit ANY file under domain/, application/, or routers/ — stop. Check whether the matching spec exists. If not, write the spec FIRST."
Sources of truth · ops knowledge

Lessons — stop re-discovering the same rake

One markdown file per non-obvious bug, named for the symptom so the next person greps and finds it. The cost is ~10 minutes; the saving is the next person not losing a day.

tilt-live-update-silently-fails

tilt trigger reports success, pod is Running, but the code in the pod is OLD.

ci-green-but-deploy-ships-broken

Image built + deployed despite a red gate — workflows raced.

runner-drops-builder-label-on-restart

A docker restart re-registers the runner with a stale env → image jobs queue forever.

Shape: When this bites · Symptoms · Root cause · Fix · How to detect early · Related. The rule is cultural, made routine: after fixing a non-obvious bug, write the lesson and link it from the commit.

Move 2 · The operating manual

CLAUDE.md is instructions, not documentation

It's the file the agent reads first and is told to obey. It doesn't describe the system — it tells the agent how to behave in it.

  • The ADR index — one grep-able line per decision.
  • Non-negotiables — layering rules, "money is a Decimal, never float", "routers register at /api/<resource>".
  • The spec-first trigger — a hard, explicit stop.
  • The enforcement-hook table — what blocks where.
  • Session habits — "every polling loop gets a timeout", "never save the same aggregate twice".

The test it must pass

Every line answers: "what should the agent DO or NOT do here?" If a line is just describing structure the code already shows, it's noise — cut it.

Kept honest by the same harness

CLAUDE.md itself is a strict-review trigger. When it drifts from reality (as it did after 5 days of parallel work), the docs-sync is a first-class task — we're literally doing that in this session's history.

Move 3 · The enforcement harness

Shift the gate left — block at the earliest boundary

CI blocks at the PR boundary. These hooks catch the same problems before you even push — cheap → costly, each blocking at its own boundary.

A gate worth calling out

Test-drift: every behavioural change carries a test

A watched source file changing with no corresponding test change in the same diff blocks the commit. It maps each source file to its test — by stem, by fuzz coverage for routers, or by bounded-context group.

  • Deliberately escapable: pure refactors, plumbing, migrations, and constants are exempt — but you say so in the commit. The point is a conscious decision per change, never silent drift.
  • Runs at commit (staged) and at push (vs the base branch), and again in CI (commit-range-scoped).
# pre-commit
== test-drift guard ==  base=main
  ✅ stem  application/place_order.py
  ❌ none  domain/order.py

⚠️  1 source change with NO test change:
    - domain/order.py

🚫 commit blocked
Add a test, or declare it test-neutral
in the commit message.
Move 4 · Isolate concurrency

One git worktree per session

The war story

Two agent sessions shared one working tree. One ran git checkout to a different branch — and the other session's uncommitted work scattered across the wrong branch and into a stash. Recovery took a full worktree reconstruction.

Fix, in layers: a launcher that starts every session in a fresh git worktree (auto-removed on exit if clean) — isolation by default. Plus a guard hook as the backstop.

# the guard fires ONLY under a live co-tenant
$ git checkout main
🚫 BLOCKED: branch-switch in a SHARED tree.
1 other live session is in this tree.
  Switching rips out their uncommitted work.
Use an isolated worktree:
  git worktree add ../repo-x -b feature-x
# solo session? never impeded. bypass once:
#   WT_ALLOW_SHARED_TREE=1
Heartbeat registry: each session touches a file recording which tree it occupies. "Co-tenant" = another live session in the same tree. Zero cost for a solo workflow.
CI/CD · the gate

The checks workflow

GitHub Actions on a self-hosted runner. One workflow, named checks, is the whole quality gate — the same discipline the local hooks enforce, run authoritatively server-side.

backend

ruff · mypy · validate_arch · check_spec_sync · test-drift · pytest (in-memory sqlite, no sidecars)

web

tsc -b (project refs incl. tests — not --noEmit) · vitest

web-perf

Lighthouse budgets against the production build — a big dep in the entry chunk goes red

Everything the local gate does, CI does again as the source of truth — so a bypassed local hook still can't merge past a red gate.

CI/CD · the deploy gate (ADR-065)

"Green CI" must actually block the deploy

The subtle failure: the image build and the test gate ran in parallel on push. A red gate still built and shipped an image. CI was green-ish and prod was broken.

Before — they race
push mainchecks  +  image-build parallel
↓
fluxdeploys the image — even if checks failed
After — image chains off checks
push mainchecks runs
↓ on workflow_run · conclusion==success
image-buildonly now — then flux deploys

Red checks → no workflow_run success → no image → no deploy. The build checks out the exact commit the gate passed on. No branch-protection plan required.

CI/CD · delivery

GitOps + immutable images

  • Flux reconciles git → cluster. The deployed state is a commit; a bump is a chore(gitops): bump images commit, reviewable and revertible.
  • Every deployed pod pins an immutable ref (digest / git-sha). No mutable :dev/:latest tags — a "rollout restart" can't silently serve stale code.
  • A runtime guard (verify-deployed-code.sh) asserts running == intended: the pod's source hash matches the working tree. Green means the cluster really runs your latest.

Why "immutable + verify"

A whole class of "it works on my machine / the pod has old code" bugs disappears when the tag can't change under you and a script proves what's actually running. Two lessons in this repo exist precisely because a mutable tag lied.
Post-deploy smoke re-checks the live pod (real cookie auth, schema-fuzz for 5xx) — the last line before "it's live and verified", said plainly.
CI/CD · layers of verification

Six kinds of test, each catching a different failure

Not "we have tests" — every layer intercepts a distinct class of problem, fast-to-slow, narrow-to-broad.

LayerWhat it catchesWhen
unitdomain logic / value objects / invariants (incl. Hypothesis properties)commit · CI
integrationUC + repo + adapter wired together — in-memory sqlite, no sidecarCI
fuzz (schemathesis)every GET endpoint stays 5xx-free on random inputCI (in-process) · post-deploy (against the real stack)
e2e (Playwright · 6 specs)SPA end-to-end flows (testid assertions, not hard-coded values)CI · post-deploy smoke
web-perf (Lighthouse)performance budget — a big dep in the entry chunk goes redCI (advisory)

The key: tests run on in-memory sqlite with no external service → fast, stable, parallelisable; real-stack verification is left to post-deploy smoke.

What test-drift binds: the first 4 rows (unit · integration · fuzz · e2e, plus frontend vitest *.test.tsx) have a source↔test mapping and are required to change together; web-perf (Lighthouse) and post-deploy smoke are not bound — they're budget / deploy gates with no per-change test file to map to.

CI/CD · code review + code scan

What humans miss, handed to machines & multiple models

Heterogeneous-LLM code review — an enforced deploy gate (ADR-076)

  • /multi-review — fans git diff main...HEAD out to three providers (Claude · Gemini CLI · Codex CLI) in parallel, then synthesises a cross-model agreement report. One model's blind spot the other two usually catch.
  • strict-review triggers: when the diff touches an aggregate/UC/router/migration/ADR/CLAUDE.md, review is no longer a suggestion — it's mandatory. You must run review and resolve every critical (or explicitly waive it) before merging.
  • How it's enforced (branch-protection required-review is 403 on this plan, so it reuses the ADR-065 lever): the verdict travels as a commit trailer REVIEW-VERDICT: pass / REVIEW-WAIVED: <reason>; a review-gate job inside checks verifies it for a strict diff → no review → checks red → no image → no deploy.
  • Two layers: the review-guard.sh PreToolUse hook blocks gh pr merge at the moment of action (fast feedback); the CI review-gate is the real wall. Honest bound: it enforces "review ran + criticals resolved/waived", not review quality.
  • /security-review — a dedicated security pass over the change.

5 code scans (security.yml)

Scangate
gitleaks (secret leak)BLOCK any secret
pip-audit / npm audit (dep CVEs)block CISA KEV only rest warn

The KEV-gating trick: block only on CVEs that are actually being exploited in the wild, warn on the rest — so vuln noise doesn't jam every merge.

DevOps · secrets design (Vault)

Secrets: one KEK, kept out of the pods that shouldn't see them, encrypted even in git

① One KEK encrypts the data

Fernet symmetric encryption, a single KEK (base64 32-byte). The KEK is injected from a k8s Secret via envFrom as MYWB_PASSWORD_VAULT_KEK — never read from .env / cmdline. Plaintext is always wrapped in SecretStr to prevent accidental logging.

② Secrets never enter the agent pod

The Warren runtime pod cannot see the KEK / OAuth / broker credentials. Data operations go through a per-session short-lived token callback back to the api (which alone holds the KEK + DB). The einvoice-scraper is the same: Google-blind, credentials passed per request.

③ GitOps secrets encrypted in git

SOPS + age encrypts secret manifests and commits them to git (version-controlled, one KEK across machines); Flux decrypts in-cluster with the age key. Secrets stop scattering out-of-band.

Identity: Google OAuth login + a server-issued session cookie (the DB stores only the token's SHA-256); X-User-Id header is a dev-only fallback, off in prod (else anyone on the public funnel could impersonate).

CI/CD · the pipeline reports on itself (ADR-077)

Every deploy leaves a one-page report card

The evidence is otherwise scattered across Actions run pages, commit trailers, and the filesystem. Collapse it into one HTML page per deployed commit — and make producing that page mandatory too.

What one report collapses

  • Cumulative inventory: counted from the working tree at that commit — ADRs / entity specs / use-case specs / lessons / bounded contexts / migrations.
  • This release: queried from the Actions API for the SHA — CI/CD workflow conclusions, the checks jobs (= gates + tests + review-gate), the security.yml jobs (= code scans).
  • Code-review verdict: read from the commit's REVIEW-VERDICT / REVIEW-WAIVED trailer (ADR-076).
  • Graceful degradation: with no gh / no network it still emits the inventory page and marks the Actions sections "not collected" rather than implying zero results.

How it's guaranteed every time

  • Producer (unconditional): post-deploy-report.yml hangs off workflow_run: checks==success (the ADR-065 deploy-authorization point) — every deploy-bound commit fires it, generating deploy/reports/<sha>.html and committing it back to main.
  • Loop-safe: the report commit carries [skip ci] → checks never runs → the workflow_run trigger can't re-fire; deploy/reports/ is also outside the image-build path filters.
  • Verifier (second signal): verify-deployed-code.sh asserts the report exists for the commit (loud warning; fatal only under MYWB_REQUIRE_RELEASE_REPORT=1, so the async race can't flake it).

Honest bound: the report is produced at deploy authorization (checks green), not at Flux-rollout completion (which has no CI-observable event) — the verifier covers the rollout-confirmation half.

A concrete example · follow one change end-to-end

"Add a credit_limit to CardAccount and surface it in the API"

The same small task, and what a developer actually hits under this harness — every point of "friction" is a guardrail.

  1. Start a session → cw add-credit-limit → you land in an isolated worktree; you can't touch anyone else's tree.
  2. You go to edit domain/card_account/entity.py → the spec-first hook nudges: write the spec first
  3. Write the spec + review with the owner: a row in the entity spec's Attributes table; the update-card.yaml inline schema follows.
  4. Implement: a Money field in domain (not float), the use case, add the field to the router's CardResponse, and a test.
  5. git commit → the commit gate: forgot the test → test-drift blocks · spec table ≠ yaml → spec-sync blocks · used a float → arch blocks. Fix, then it passes.
  6. (If you skipped the worktree and a co-tenant is live) a git checkout to compare branches → worktree guard blocks, tells you to use a worktree.
  7. push → PR → CI runs the full suite (pytest · tsc -b · vitest).
  8. merge → checks on main go green → image build → Flux deploys → verify-deployed-code.sh + smoke confirm the live pod really is this commit.

The experience: each gate catches you before you've gone far wrong, and its message tells you how to fix it — friction as guardrail.

Apply it to your project

It's packaged — one command

The generic layer is extracted into claude-control-kit (a repo) + a /scaffold-controls skill. Install the harness into any repo; fill in what's yours.

# base harness (any language)
$ bash claude-control-kit/install.sh
  ✓ worktree guards + launcher
  ✓ commit/push gate runner (.dev/gates/*)
  ✓ .dev/ scaffold (adr/lessons/tasks/gates)
  ✓ CLAUDE.md + CI + settings templates
  # idempotent · never clobbers existing files

# + full Python/DDD ruleset as a profile
$ bash install.sh --profile python-ddd
  ✓ ruff · mypy · 22-rule arch lint
  ✓ spec-sync · test-drift · spec-first hook

Generic vs yours — the split

Ships working: worktree isolation, the gate wiring, the .dev/ discipline, the CI deploy-gate pattern.
You fill in: the actual gates — drop executable scripts in .dev/gates/ (your lint, types, test-drift map, arch rules). The runner is language-agnostic; what you enforce is yours.
Verified: the arch linter, re-parameterized, reproduces the reference project's own result — and blocks a real layering violation on a different package. The kit self-tests 6/6.
Be honest about it

What the harness actually enforces

Three tiers — know which is which so you set expectations, not surprises.

auto · out of the box

Concurrency safety

Multi-session git checkout/stash/reset is hard-blocked under a co-tenant. Zero config; dormant for a solo session.
opt-in · you fill

Commit/push gates

The runner is wired but empty until you add gates to .dev/gates/. Then lint/type/arch/test-drift hard-block — humans and agents alike.
advisory

Nudges + conventions

Push-to-main warning, co-tenant edit warning, and the .dev/ methodology + CLAUDE.md — enforced by convention + review, not a blocker.

The kit gives you a governance framework + one concurrency safety net, not a canned rulebook. The rules should be yours; the harness gives them teeth.

Getting started

Adoption checklist

  1. Run install.sh in your repo (add --profile python-ddd if it fits). Idempotent, non-destructive.
  2. Activate git hooks: git config core.hooksPath .githooks. Merge the hooks block into .claude/settings.json.
  3. Fill .dev/gates/ with your real lint / typecheck / test-drift / arch gates. Confirm each blocks on a deliberate violation, then passes.
  4. Write your CLAUDE.md from the template — stack, conventions, and start an ADR index (ADR-000 ships as the template).
  5. Wire CI: a checks workflow; if you deploy on push, adopt the workflow_run deploy-gate so a red gate blocks delivery.
  6. Add the cw launcher to your shell rc; start sessions with it so each gets an isolated worktree.
  7. Then let it grow: every incident → a new gate or a lesson. The system hardens itself.
Takeaways

Seven transferable principles

  • Sources of truth live in the repo — decisions, behaviour, lessons, work state. Versioned, greppable, next to the code.
  • Give the agent an operating manual — instructions it must follow, not prose it might read.
  • Spec-first — the design is reviewed before the code, and a checker keeps them from drifting.
  • Shift enforcement left — block at edit/commit/push, not just at the PR. Redundant layers on purpose.
  • Every behavioural change carries a test — no silent test drift.
  • Green CI must block the deploy — chain the build off the gate, not the push.
  • Isolate concurrency — a worktree per session; and let each incident harden into a gate or a lesson.
